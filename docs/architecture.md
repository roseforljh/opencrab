# OpenCrab 架构说明

## 1. 文档目的

这份文档用于固定 OpenCrab 首版的整体架构，避免后续在开发过程中反复摇摆。

它回答 4 个问题：

1. 这个项目是什么。
2. 这个项目首版做什么。
3. 这个项目由哪些部分组成。
4. 这些部分之间如何协作。

## 2. 项目定位

OpenCrab 是一个面向个人部署的大模型聚合 API 网关与管理台。

它的核心目标是：

1. 统一接入多个上游模型渠道。
2. 提供 OpenAI 兼容接口。
3. 提供基础管理台，方便配置与查看。
4. 保持轻量、稳定、易部署、可扩展。

## 3. 首版边界

### 3.1 首版要做的内容

1. OpenAI 兼容接口。
2. OpenAI 兼容 provider 接入。
3. 普通响应与 SSE 流式响应。
4. 渠道配置管理。
5. 模型映射与路由。
6. API Key 管理。
7. 基础限流。
8. 请求日志摘要。
9. 健康检查与就绪检查。
10. 基础管理台。

### 3.2 首版不做的内容

1. 商业计费。
2. 钱包、充值、结算。
3. 多租户。
4. 复杂 RBAC。
5. 分布式集群。
6. 插件系统。

## 4. 总体架构

首版采用“单仓库、前后端分离、单机优先”的架构。

### 4.1 后端

后端负责：

1. 接收客户端请求。
2. 校验访问权限。
3. 根据模型映射与路由规则选择上游渠道。
4. 转发请求到上游 provider。
5. 处理普通响应或流式响应。
6. 记录日志摘要。
7. 提供管理端 API。

### 4.2 前端

前端负责：

1. 提供管理台界面。
2. 展示系统状态。
3. 管理渠道、模型、密钥、日志、系统设置。

### 4.3 数据存储

首版默认使用 SQLite。

SQLite 用于保存：

1. 渠道配置。
2. 模型映射。
3. 路由规则。
4. API Key。
5. 请求日志摘要。
6. 系统设置。

## 5. 后端内部层次

后端按职责分层，避免代码混在一起。

### 5.1 `transport/http`

负责 HTTP 协议相关内容。

包括：

1. 路由注册。
2. 请求解析。
3. 响应输出。
4. 中间件。

### 5.2 `usecase`

负责业务编排。

包括：

1. 代理请求流程。
2. 渠道管理流程。
3. 模型路由流程。
4. API Key 管理流程。
5. 日志查询流程。

### 5.3 `provider`

负责上游模型供应商适配。

包括：

1. 构造上游请求。
2. 解析上游返回。
3. 适配流式输出。
4. 转换错误格式。

### 5.4 `store`

负责和数据库交互。

包括：

1. 查询。
2. 写入。
3. 更新。
4. 删除。

### 5.5 `config`

负责配置读取、默认值和校验。

### 5.6 `observability`

负责日志、request id、错误记录等可观测能力。

## 6. 核心请求链路

一次典型代理请求的流转顺序如下：

1. 客户端调用 OpenAI 兼容接口。
2. HTTP 层解析请求并做基础校验。
3. API Key 中间件校验访问身份。
4. usecase 层根据模型名查找映射关系。
5. usecase 层根据路由规则选择 provider。
6. provider 层把统一请求转换成上游请求。
7. 后端通过共享 HTTP 客户端访问上游。
8. 上游返回普通响应或流式响应。
9. 系统将返回结果转换为统一输出格式。
10. 系统记录日志摘要并返回给客户端。

## 7. 前端架构

前端采用 Next.js + Tailwind CSS + shadcn/ui。

前端核心页面包括：

1. Dashboard。
2. Channels。
3. Models & Routing。
4. API Keys。
5. Logs。
6. Settings。

前端设计方向采用 Vercel / Geist 风格，强调：

1. 开发者工具气质。
2. 信息密度高。
3. 表格、日志、表单清晰。
4. 中文界面克制、整洁。

## 8. 非功能要求

首版从一开始就要满足这些要求：

1. 支持超时控制。
2. 支持取消传播。
3. 支持基础限流。
4. 日志默认脱敏。
5. 配置必须校验。
6. 流式请求中断后能正确释放资源。
7. 本地和 Docker 都能部署。

## 9. 当前结论

OpenCrab 首版不追求功能面铺开，而是优先把“前端管理台整体体验”先做完整，等前端确认满意后，再继续把后端代理主链路、配置管理和日志能力打稳。
