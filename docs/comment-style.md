# OpenCrab 注释规范

## 1. 目标

这个项目要求所有注释统一使用中文，而且要尽量详细，让不了解代码的人也能看懂当前模块在干什么。

注释不是为了凑数量，而是为了把“这段代码为什么存在、在做什么、和谁配合、出错会怎样”说清楚。

## 2. 总规则

1. 所有注释统一使用中文。
2. 所有重要文件都必须有文件头注释。
3. 所有结构体、接口、函数都必须有中文说明。
4. 所有复杂逻辑必须有步骤性注释。
5. 所有配置项必须有中文说明。
6. 所有数据库表与字段必须有中文说明。

## 3. 文件头注释规范

每个重要文件开头都要写一段中文块注释，至少说明：

1. 这个文件负责什么。
2. 这个文件依赖什么。
3. 这个文件会被谁使用。
4. 这个文件的输出结果是什么。

## 4. 结构体与接口注释规范

每个结构体和接口都要解释它在业务里的含义。

例如：

1. 这个结构体表示什么业务对象。
2. 这个接口是给哪一层用的。
3. 为什么要抽这个接口。

## 5. 函数注释规范

每个公开函数都必须写中文注释，尽量包含：

1. 这个函数做什么。
2. 输入参数分别代表什么。
3. 返回值代表什么。
4. 什么时候会失败。

## 6. 复杂逻辑注释规范

凡是下面这些情况，都必须在代码块前写步骤性注释：

1. 请求转发。
2. 流式响应处理。
3. 错误转换。
4. 路由选择。
5. 配置回退。
6. 限流判断。

推荐写法：

1. 先说明当前步骤要解决的问题。
2. 再说明为什么这么处理。
3. 最后说明失败时会怎样。

## 7. 配置注释规范

每个配置项都要说明：

1. 配置项用途。
2. 默认值。
3. 配错后的影响。

## 8. 数据库注释规范

每个表和字段都要能让人看懂业务意义。

至少说明：

1. 这张表存什么。
2. 这个字段表示什么。
3. 这个字段为什么存在。

## 9. 禁止事项

1. 禁止写无意义注释。
2. 禁止写和代码完全重复的空注释。
3. 禁止中英混乱、术语不统一。
4. 禁止复杂代码没有解释。

## 10. 结果要求

最终效果应满足：

1. 不看实现细节，也能知道模块职责。
2. 不熟悉 Go 或 Next.js，也能大致看懂流程。
3. 后续维护时，不需要靠猜来理解关键逻辑。
