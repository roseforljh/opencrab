# OpenCrab 首版详细计划总纲

## 1. 项目定位

OpenCrab 是一个面向个人部署的大模型聚合 API 项目。

它的目标不是做商业化平台，也不做充值、计费、分销、多租户，而是做一个基础扎实、性能稳定、结构清晰、后续容易扩展的个人自托管网关与管理台。

首版核心原则：

1. 先打基础，不追求功能数量。
2. 当前开发顺序改为先把前端管理台完整做出来，再进入后端实现。
3. 先保证个人部署体验，再考虑扩展增强。
4. 所有代码、注释、文档统一中文。

## 2. 首版明确要做什么

首版只做以下能力：

1. 多上游渠道接入。
2. OpenAI 兼容接口。
3. 模型映射与路由。
4. 普通响应与流式响应。
5. 管理台配置与查看。
6. API Key 管理。
7. 基础限流。
8. 日志摘要与健康检查。

## 3. 首版明确不做什么

首版不做以下内容：

1. 商业计费。
2. 钱包、充值、结算。
3. 多租户。
4. 复杂权限系统。
5. 分布式集群。
6. 插件市场。
7. 复杂工作流引擎。
8. 过早引入 Redis、MQ、Prometheus 全家桶作为强依赖。

## 4. 技术方案

### 后端

- 语言：Go
- Web 框架：优先 `chi`，如后续需要更高开发效率可评估 `gin`
- HTTP 能力：标准库 `net/http` + 共享 `Transport`
- 数据库：SQLite
- 日志：结构化日志
- 配置：配置文件 + 环境变量覆盖

### 前端

- 框架：Next.js
- 样式：Tailwind CSS
- 组件：shadcn/ui + Radix UI
- 表格：TanStack Table
- 表单：React Hook Form + Zod
- 视觉方向：以 Vercel / Geist 风格为主，面向开发者控制台

## 6. 首版数据库对象

首版数据库尽量少实体，只保留必要对象：

1. `channels`：上游渠道。
2. `models`：模型映射。
3. `model_routes`：路由规则。
4. `api_keys`：访问密钥。
5. `request_logs`：请求日志摘要。
6. `system_settings`：系统设置。

## 7. 关键基础能力

第一版必须从一开始就具备这些基础能力：

1. 统一 provider 适配接口。
2. OpenAI 兼容层。
3. SSE 流式输出链路。
4. 超时、取消传播、重试上限。
5. request id。
6. 限流与并发控制。
7. 结构化日志与错误码。
8. 敏感字段脱敏。
9. 健康检查与就绪检查。
10. 配置校验与默认值控制。

## 7.1 首个 provider 决策

首版第一个必须接入的 provider 固定为：`OpenAI 兼容渠道`。

原因：

1. 它是聚合网关最核心、最通用的兼容目标。
2. 后续接入其他供应商时，可以先统一对齐到 OpenAI 风格的输入输出。
3. 首版先把 OpenAI 兼容普通响应与 SSE 流式响应打稳，后面再扩 Anthropic、Gemini、其他兼容供应商，风险最小。

首版阶段 2 的具体边界固定为：

1. 只要求打通一个 OpenAI 兼容 provider。
2. 只要求优先完成 `/v1/chat/completions`。
3. 普通响应和 SSE 流式响应都必须可用。
4. 暂不在阶段 2 引入多 provider 并发竞速与复杂回退链。

## 9. 注释规范

整个项目统一执行这些规则：

1. 每个文件头部必须写中文块注释，说明用途、依赖、输出对象。
2. 每个结构体必须写中文说明。
3. 每个函数必须写中文说明，包含用途、参数、返回值、异常情况。
4. 每个复杂逻辑前必须写步骤性中文注释。
5. 每个配置项必须有中文说明。
6. 每个数据库表与字段必须有中文说明。

## 10. 阶段计划总览

### 阶段 0：方案固化

目标：把规则、边界、目录、注释规范先固定下来。

### 阶段 1：前端管理台基础

目标：建立 Next.js 管理台壳子、全局布局、设计 token 和基础组件体系。

### 阶段 2：前端核心页面

目标：完成 Dashboard、Channels、Models、API Keys、Logs、Settings 页面主体。

### 阶段 3：前端联调前完整度提升

目标：补齐页面细节、空状态、抽屉、筛选、假数据流与统一组件封装，让前端达到可验收状态。

### 阶段 4：后端基础骨架

目标：在用户确认前端满意后，再开始服务启动、配置读取、数据库初始化与健康检查。

### 阶段 5：代理主链路

目标：打通 OpenAI 兼容接口与第一个 provider 的普通响应和流式响应。

### 阶段 6：后端管理能力

目标：完成渠道管理、模型路由、API Key、日志、限流等后端能力。

### 阶段 7：部署与验收

目标：在前后端都完成后，统一做 Docker、部署文档、验收清单。

## 10.1 阶段验证原则

每个阶段都必须带验证，不允许只写“完成开发”。

统一规则：

1. 每个阶段都要明确验证工具。
2. 每个阶段都要明确验证输入。
3. 每个阶段都要明确预期结果。
4. 每个阶段都要能让后续执行者直接照着验证。

验证工具优先级：

1. 命令行接口验证。
2. curl / HTTP 请求验证。
3. 数据库查询验证。
4. 页面手动操作验证。
5. Docker 部署验证。

## 11. 实际执行顺序

严格按下面顺序推进：

1. 先写文档。
2. 先把前端壳子做完整。
3. 再把前端六个核心页面做完整。
4. 再把前端细节、组件封装和交互补完整。
5. 等用户确认前端满意后，再开始后端。
6. 最后做部署与验收。

## 12. 当前推荐下一步

正式开工时，第一批动作固定为：

1. 创建核心文档。
2. 初始化 Next.js 项目骨架。
3. 搭完整体布局和导航系统。
4. 完成六个前端核心页面。
5. 完成前端统一组件封装。
6. 等前端确认后，再开始后端。
